# 實測發現與優化方向

> 基於四份實測報告（行為經濟學、Anthropic 認證、Threads API、GTM + JSON-LD）的交叉分析。
> 目的：確認產品定位、辨識強弱項、提出具體優化方向。

---

## 1. 四份實測概覽

| | 行為經濟學 | Anthropic 認證 | Threads API | GTM + JSON-LD |
|---|---|---|---|---|
| **領域類型** | 社會科學 | 科技＋倫理＋商業 | 純技術規範 | 純技術規範 |

| **深度/爭議性** | 高（百年學術辯論） | 極高（版權＋軍事＋AI末日） | 低（單向技術規範） | 低（技術規範＋選型比較） |

## 3. 五個核心發現

### 發現 1：Q4（第一原則）是殺手功能

四次測試都精準歸納出 3 條原則，且都達到「記住這 3 條就能推導整個領域」的效果：

| 領域 | 三條第一原則 |
|------|------------|
| 行為經濟學 | 大腦有極限 → 怕失去 → 被包裝騙 |
| Anthropic | 輸入決定品質 → 能力需要約束 → 大腦需要手腳 |
| Threads API | 拿對鑰匙 → 大包裹先寄存 → 遵守限速 |
| GTM + JSON-LD | 機器沒常識要貼標籤 → 資料動態需解耦 → 幫機器省力換 VIP 版面 |

**為什麼是殺手功能：**
- P1（焦慮職場人）最需要的就是「一句話說清楚這個領域在幹嘛」
- Q4 產出的 3 條原則，直接可以拿來在會議上發言、寫報告
- 這是最容易變成 Threads 貼文素材的產出

### 發現 2：Q6 分支策略驗證成功

**原始問題：** Q6-A（核心分歧）在技術文檔場景空轉。

| 資料性質 | Q6-A 表現 | Q6-B 表現 |
|----------|-----------|-----------|
| 學術百科（Wikipedia） | ★★★★★（天然多元觀點） | —（未測） |
| 混合來源（官方＋第三方＋百科） | ★★★★★（多角度覆蓋） | —（未測） |
| 純官方文檔（Threads API） | ☆☆☆☆☆（空轉） | —（未測） |
| 技術文檔＋教學（GTM + JSON-LD） | —（未測） | ★★★★★（2 組完整比較） |

**Q6-B 首測結論：** 在技術領域，Q6-B 產出 2 組清晰的替代方案比較（JSON-LD vs Microdata、SSR vs GTM 動態生成），每組都有完整的優缺點、適用場景、業界傾向。**Q6 的空轉問題已被 Q6-B 解決。**

### 發現 3：資料策劃（Source Curation）決定天花板

三次測試的最大品質差異不在提問，而在資料來源：

- **行為經濟學**：Wikipedia 天然包含「批評」章節 → Q6 爆發
- **Anthropic**：混合來源（學習平台 + 官方文檔 + 百科 + 新聞）→ 全面且平衡
- **Threads API**：純官方文檔 → Q1-Q5 強，Q6-A 空轉
- **GTM + JSON-LD**：官方＋百科＋教學 → Q1-Q5 強，Q6-B 替代方案比較爆發

**結論：「怎麼選資料」應該是工作流的核心指引，而不是附帶說明。資料組合也影響 Q6 版本的選擇。**

### 發現 4：認知階梯效應是真實的

Q1→Q2→Q3→Q4→Q5→Q6→Q7 的漸進順序不是裝飾——每一題都在前一題的基礎上深入：

```
Q1（術語）→ 知道「有什麼」
Q2（因果）→ 知道「怎麼串」
Q3（現象）→ 知道「會發生什麼」
Q4（原則）→ 知道「為什麼會這樣」
Q5（框架）→ 知道「專家怎麼想」
Q6（分歧）→ 知道「哪裡還沒定論」
Q7（結構）→ 把以上全部串成地圖
```

這種漸進式披露是工作流最大的結構性優勢——使用者不需要自己想問什麼，跟著問就對了。

### 發現 5：40 分鐘的認知跳躍可被驗證

四次 Before→After 都有明確的認知落差：

| 領域 | Before | After |
|------|--------|-------|
| 行為經濟學 | 聽過「損失規避」但串不起來 | 能畫出認知偏誤的因果網路 + 三大學術爭議 |
| Anthropic | 知道 AI 安全很重要 | 能分析版權/軍事/末日三大爭議的框架假設衝突 |
| Threads API | 不知道怎麼申請 Meta API | 四層知識地圖 + 三條第一原則 |
| GTM + JSON-LD | 用過 GTM 但不懂底層機制 | 五層知識地圖 + 能做 JSON-LD vs Microdata 選型 |

## 4. 產品定位確認

### 4.1 核心承諾

> **「1 小時內，從『完全沒聽過』到『能跟這行的人聊』」**

三份實測確認：**這個承諾是可兌現的。**

Bloom's Taxonomy（布魯姆分類法）覆蓋：

> Bloom's Taxonomy 是教育心理學的經典框架，把「學會了」分成六個層級：
> 記住（背得出來）→ 理解（說得出來）→ 應用（做得出來）→ 分析（拆得開）→ 評價（判得了）→ 創造（造得出）。
> 我們的工作流覆蓋到「分析」，碰不到「應用」以上——**學完能聊但不能做**。
> 對 P1 來說這剛好夠用，因為他們要的是「開會時聽得懂、能發言」。

```
Remember  ✅（Q1 術語）
Understand ✅（Q2-Q3 因果＋現象）
Analyze   ✅（Q4-Q7 原則＋框架＋結構）
Apply     ⚠️（Q6-B 觸及邊緣 — 「什麼情況該選哪個」）
Evaluate  ⚠️（Q6-A 有條件涵蓋）
Create    ❌（工作流不涵蓋）
```

### 4.2 最佳適用場景

| 場景 | 適合度 | 說明 |
|------|:------:|------|
| 學術/概念性領域 | ★★★★★ | 全 7 題滿分，Q6 爆發 |
| 混合型領域（商業＋技術＋倫理） | ★★★★★ | 全 7 題滿分 |
| 技術規範型領域 | ★★★★★ | Q1-Q5 + Q7 強，Q6-B 替代方案比較有效（已驗證） |
| 考試準備 | ★★☆☆☆ | 概念理解強，操作細節弱 |
| 實作技能 | ★☆☆☆☆ | 不涵蓋 Apply 層 |

### 4.3 目標受眾匹配度

| Persona | 匹配度 | 理由 |
|---------|:------:|------|
| P1 焦慮職場人 | ★★★★★ | 他們要的就是「能聽懂、能發言」，完美契合 |
| P2 內容創作者 | ★★★★☆ | Q4 + Q7 是絕佳寫文骨架，但缺 Apply 層 |
| P3 轉職探索者 | ★★★☆☆ | 概念偵查有效，但他們真正需要的是實戰經驗 |
| P4 知識愛好者 | ★★★★☆ | 結構化理解正是他們想要的 |

### 4.4 產品交付形式：雙軌

產品不只是 Claude Code skill——**prompt 套組本身就是產品**。兩種交付形式服務不同的使用者：

| | Prompt 套組 | Skill |
|---|---|---|
| **是什麼** | 一組可直接複製貼上的提問句 + 使用指南 | Claude Code 自動化工作流 |
| **使用者** | 所有 NotebookLM 使用者 | Claude Code + notebooklm-py 使用者 |
| **門檻** | 零（打開 NotebookLM → 貼上 → 問） | 需要 Claude Code + notebooklm-py |
| **市場大小** | 大（任何 NotebookLM 使用者） | 小（開發者/進階使用者） |
| **自動化** | 手動逐題操作 | Agent 自動跑完全流程 |
| **客製化** | 使用者自己判斷選哪題 | Agent 根據訊號自動選 |
| **分發管道** | Threads 貼文、部落格、GitHub | npx skills add、GitHub |

**關鍵認知：** P1（焦慮職場人）和 P4（知識愛好者）幾乎不可能用 Claude Code。他們的路徑是：

```
看到 Threads 貼文 → 「這方法好像有用」
→ 點進連結看 prompt 套組 → 打開 NotebookLM
→ 上傳材料 → 照順序貼上提問句 → 完成
```

這意味著 **prompt-templates.md 的使用體驗就是產品體驗**。它不是給 agent 看的參考檔——它是使用者直接面對的介面。

#### 雙軌對產品設計的影響

| 設計決策 | Prompt 套組（手動） | Skill（自動） |
|----------|:--:|:--:|
| Q6 決策樹 | 使用者讀一句話判斷，自己選 A 或 B | Agent 根據訊號自動選 |
| 資料策劃 | 使用者看 3 分鐘檢查清單 | Agent 主動檢查並建議補充 |
| Q7 連動 | 使用者根據自己選的 Q6 版本，貼對應的 Q7 | Agent 自動連動 |
| 生成操作 | 使用者手動點 NotebookLM 介面的按鈕 | Agent 用 CLI 執行 |
| 收尾摘要 | 使用者自己回顧 | Agent 自動生成 |

#### 對現有文件的影響

| 文件 | 現在的角色 | 應該的角色 |
|------|-----------|-----------|
| prompt-templates.md | Agent 的提問句參考 | **面向使用者的產品介面**（prompt 套組的主體） |
| user-guide.md | 方法論說明 | prompt 套組的「為什麼這樣問」說明 |
| SKILL.md | Agent 執行協議 | 不變（只給 agent 看） |

**最重要的改動：prompt-templates.md 必須按「手動使用者能直接照做」的標準重新設計。** 加入資料策劃檢查清單、Q6 二選一提示、Q7 連動提示——這些不是附加功能，而是產品的一部分。

## 5. 優化方向

### 5.1 Q6 分支邏輯（優先度：高 → ✅ 已驗證）

**問題：** Q6-A 在技術文檔場景空轉。

**方案：** Q6-B（替代方案比較）已在 GTM + JSON-LD 案例中首測成功。根據領域類型用決策樹判斷。

#### Q6 決策樹

```
你要學的東西是什麼類型？
│
├─ 有「學派之爭」「觀點對立」的領域嗎？
│   │  （經濟學的學派之爭、AI 的安全 vs 能力、
│   │    醫學的治療方案辯論⋯⋯）
│   │
│   ├─ YES → Q6-A：核心分歧
│   │   │
│   │   │  「專家們在哪些核心議題上有根本性的分歧？
│   │   │   請列出 2-3 個最重要的爭議，每個告訴我：
│   │   │   1. 爭議的焦點是什麼
│   │   │   2. 正方的觀點和主要證據
│   │   │   3. 反方的觀點和主要證據
│   │   │   4. 目前學界比較傾向哪邊」
│   │   │
│   │   │  學到：這個領域有哪些未定論
│   │   │  能力層級：Analyze
│   │   │
│   │   └─ 適用：社會科學、人文、商業策略、政策辯論、
│   │         新興科技倫理（AI安全、基因編輯⋯）
│   │
│   └─ NO → 有「多種做法」「替代方案」可以比較嗎？
│       │  （API 文檔、框架選型、工具指南、
│       │    程式語言、流程設計、商業決策⋯⋯）
│       │
│       ├─ YES → Q6-B：替代方案比較
│       │   │
│       │   │  「在這個領域中，有哪些地方存在不同的做法或替代方案？
│       │   │   請列出 2-3 個最重要的選擇，每個告訴我：
│       │   │   1. 有哪些替代方案（或不同的做法）
│       │   │   2. 各自的優缺點
│       │   │   3. 什麼情況下該選哪個
│       │   │   4. 目前業界比較傾向哪個（如果有的話）」
│       │   │
│       │   │  學到：不同方案的能力邊界，什麼情況用什麼
│       │   │  能力層級：Analyze + Apply 邊緣
│       │   │
│       │   └─ 適用：技術選型、工具比較、框架學習、
│       │         流程設計、商業決策、產品策略
│       │
│       └─ NO → Q6-A + Q6-B 都問
│           │
│           └─ 適用：不確定的領域、混合型內容
│               （例如「區塊鏈」既有技術規範也有治理爭議）
```

#### Q6-A vs Q6-B 定位

Q6-B 不是「Q6-A 不能用時的降級替代」——兩者是平等的，服務不同類型的知識。

| | Q6-A（核心分歧） | Q6-B（替代方案比較） |
|---|---|---|
| 問什麼 | 誰跟誰吵 | 怎麼選 |
| 結構 | 爭議焦點 / 正方 / 反方 / 學界傾向 | 替代方案 / A 優缺 / B 優缺 / 業界傾向 |
| 學到什麼 | 這領域有哪些未定論 | 不同方案的能力邊界 |
| 能力層級 | Analyze | Analyze + Apply 邊緣 |
| 最適合 | 學術/概念/政策/倫理 | 技術/工具/流程/商業決策 |
| P1 的場景 | 「我知道這件事有爭議」 | 「老闆問我選哪個，我能回答」 |

#### 判斷指引

**Agent 場景（SKILL.md 指引）：**

Agent 根據以下訊號自動選擇：

| 訊號 | 指向 Q6-A（分歧） | 指向 Q6-B（比較） |
|------|:--:|:--:|
| 資料來源包含 Wikipedia | ✅ | |
| 資料來源包含評論/觀點文章 | ✅ | |
| 資料來源全部是官方文檔 | | ✅ |
| 資料來源包含教學/tutorial | | ✅ |
| Q5 框架回答中出現「學派」「流派」「陣營」 | ✅ | |
| Q5 框架回答中出現「最佳實踐」「建議做法」 | | ✅ |
| 學習主題涉及「選型」「比較」「該用哪個」 | | ✅ |
| 不確定 | 兩題都問 | 兩題都問 |

**非 Agent 場景（使用者自己操作 NotebookLM）：**

在 prompt-templates.md 中加入簡單判斷提示：

> **Q6 有兩個版本，選適合你的：**
>
> - 你在學的東西**有人在吵**嗎？（學派之爭、政策辯論、價值觀衝突⋯）
>   → 用 **Q6-A**（核心分歧）— 了解這領域還沒定論的地方
>
> - 你在學的東西**有多種做法可以比較**嗎？（技術選型、工具比較、流程設計⋯）
>   → 用 **Q6-B**（替代方案比較）— 了解什麼情況該用什麼
>
> - 不確定？兩個都問，NotebookLM 會自己判斷哪個答得出來。

#### Q6 → Q7 連動

Q7 接在 Q6 後面，措辭要跟著 Q6 版本調整：

| Q6 版本 | Q7 問法 |
|---------|---------|
| Q6-A（分歧） | 「這些框架和**分歧**之間的關係是什麼？哪些互補？哪些互斥？哪些分歧源自不同的框架假設？」 |
| Q6-B（比較） | 「這些框架和**概念**之間的關係是什麼？哪些是互補的？哪些是有先後依賴的？請幫我理清它們的結構，畫出一個完整的知識地圖。」 |

在 prompt-templates.md 中，Q7 也提供 A/B 兩版讓使用者對應選擇。

### 5.2 資料策劃指引（優先度：高）

**問題：** 資料品質直接決定學習天花板，但工作流沒有指導使用者怎麼選資料。垃圾進垃圾出——這個規律在四次實測中都被驗證了。

**核心原則：** 好的資料組合 = 「事實 + 觀點 + 全貌」三角形。缺任何一角，學習都會有盲區。

#### 理想的資料組合（5-7 篇）

| 角色 | 數量 | 做什麼用 | 範例 |
|------|:----:|----------|------|
| 全貌型 | 1-2 篇 | 提供整個領域的鳥瞰圖 | Wikipedia、教科書導論、「XX 入門指南」 |
| 權威型 | 2-3 篇 | 提供準確的定義和細節 | 官方文檔、經典論文、RFC、技術規格 |
| 觀點型 | 1-2 篇 | 提供多元視角和爭議 | 評論文章、專家訪談、爭議報導、部落格觀點文 |

#### 常見錯誤

| 錯誤 | 後果 | 實測證據 |
|------|------|----------|
| 全部官方文檔 | Q6-A 空轉（無觀點碰撞）| Threads API：Q6-A 回「無分歧」；GTM：改用 Q6-B 產出 2 組比較 |
| 全部入門科普 | Q4 底層邏輯太淺 | — |
| 內容高度重複 | 浪費 source 額度，不增加資訊量 | — |
| 只放一篇超長文 | NotebookLM 可能只聚焦前半段 | — |

#### 兩種使用者場景

**場景 A：有 Agent 幫忙（Claude Code + notebooklm-py）**

Agent 可以主動幫使用者搜集資料：

```
1. 用 WebSearch 搜「[領域] overview」「[領域] controversy」「[領域] tutorial」
2. 用 WebFetch 或 crawl4ai 抓取 3-5 篇
3. 確認三角形是否完整（全貌 + 權威 + 觀點）
4. 用 notebooklm source add 上傳
```

SKILL.md 應加入指引：在上傳資料前，agent 檢查資料組合是否覆蓋三角形。如果缺「觀點型」，主動建議補一篇 Wikipedia 或評論文章。

**場景 B：使用者自己操作（手動使用 NotebookLM）**

這是多數使用者的情況——他們不用 Claude Code，直接在 NotebookLM 網頁介面操作。

user-guide.md 和 prompt-templates.md 需要加入簡單明瞭的指引：

> **上傳材料前，花 3 分鐘做這個檢查：**
>
> ✅ 有沒有一篇「告訴你全貌」的文章？（Wikipedia 是最簡單的選擇）
> ✅ 有沒有官方或權威來源？（教科書、官方文檔、經典論文）
> ✅ 有沒有「有人在吵」的文章？（評論、觀點文、爭議報導）
>
> 三個都有 → 你的學習效果會最好。
> 只有官方文檔 → Q6-A（分歧辨識）會空轉，改用 Q6-B（替代方案比較）。
> 只有科普 → Q4（底層邏輯）可能太淺，建議補一篇深度文章。

#### NotebookLM 內建的 Research 功能

NotebookLM 有「add research」功能，可以自動搜集相關材料。這對非 agent 使用者特別有用：

> 在 NotebookLM 介面中，點「Add source」→「Research」→ 輸入關鍵字。
> NotebookLM 會自動找到相關的網路資源加入你的筆記本。

這個功能可以彌補使用者不知道去哪找「觀點型」資料的問題。建議在 user-guide.md 提醒：上完自己的資料後，再用 Research 功能補 1-2 篇。

### 5.3 Q7 產出強化（優先度：中）

**現況：** Q7 已經會自動適應領域類型（辯證結構 vs 依賴地圖），但目前只是文字描述。

**方案：** 在 Q7 之後加入 mind-map 生成步驟，將 Q7 的結構視覺化。目前 mind-map 是獨立生成的，沒有利用 Q7 的結構化產出。

### 5.4 After 產出模板（優先度：中）

**問題：** 學完之後的「產出」沒有結構。P2（內容創作者）需要「學完就能寫文章」。

**方案：** 加入可選的第 8 步——「基於你剛才學到的，幫我草擬一篇 XX 字的結構化摘要/文章骨架」。利用 Q4（原則）+ Q7（結構圖）作為骨架，Q6（分歧）作為觀點差異化的素材。

### 5.5 技術文檔專用問題（優先度：低）

**問題：** Q3（重要現象）在技術領域產出偏少（4 個 vs 學術的 11 個），因為技術文檔的「現象」本質上是「機制」而非「效應」。

**方案：** 技術場景可替換為「這個技術中有哪些關鍵機制或設計模式？每個機制告訴我它解決什麼問題、怎麼運作、什麼時候會遇到它。」

## 6. 行銷素材方向（基於實測數據）

### 6.1 最有說服力的 hook

基於四次測試，最能打動 P1 的 hook：

> 「我花 1 小時用 NotebookLM 搞懂了 [領域]，歸納出 3 條核心原則——記住這 3 條，整個領域的知識都能推導出來。」

Q4 產出是最適合做 Threads 貼文的素材，因為：
- 簡潔（永遠是 3 條）
- 有「頓悟感」（看完會覺得「原來如此」）
- 可驗證（讀者可以自己跑一次看是不是真的）

### 6.2 Before/After 對比素材

Threads API 案例最適合做行銷素材：
- Before：「我不知道怎麼申請 Meta API」
- After：「我能畫出四層知識地圖，告訴你 OAuth 認證→通訊架構→發布模型→防禦機制的完整依賴關係」
- 時間：40 分鐘
- 額外產出：mind-map + quiz + podcast

這個落差夠大、夠具體、而且學習主題（Threads API）本身就是 Threads 使用者會感興趣的。

### 6.3 系列貼文結構

```
第 1 篇：hook — Before/After 落差展示（Threads API 案例）
第 2 篇：方法 — 7 步認知階梯是什麼、為什麼有效
第 3 篇：實證 — Q4 的 3 條原則展示（選一個大眾有感的領域）
第 4 篇：教學 — 「你也來試試」+ 工作流連結
```
